Garmin's first wireframing kit for wearables
Timeline 12 months(2024 - 2025)
Project Role Lead Architect
Responsibilities Led research on design systems and met with other teams to discuss opportunities for collaboration and future growth. Defined the basic vision and goals for the project and set up processes for tracking existing UI and updating the library. Created library elements and the documentation to support them, then compiled a handoff document to help onboard new users and future architects.

Wearables are a major part of Garmin’s product portfolio, boasting 6+ regularly updated product families and making up nearly 40% of the company’s multi-billion dollar revenue stream.

Prior to the development of this library, every product designer worked in isolation on one product line. This approach began to fail as Garmin’s offerings grew and diversified. Designers struggled to create cohesive experiences across screen sizes and features, and onboarding new team members became a nightmare of a task.

I was asked to build a wireframing kit that could address these bottlenecks, and set out to build a system that could scale rapid development and seamless onboarding across diverse product families.

Defining the problem

I began by studying other design systems and learning more about the Wearable team’s day-to-day pain points.

There were two core issues:

  1. Each designer had to figure out how to build common elements and interactions on their own. This made mocking up simple screens and flows extremely time-consuming and laborious. As a result...
  2. Designers often resorted to using time-saving shortcuts.
    Strategies included using edited screenshots, copying elements from related projects, and creating their own ad-hoc components.

This had several cascading effects.

  • Outdated design patterns quickly circulated and were difficult to track down and validate.
  • Converting features between product lines (a very common task) took an inordinate amount of time to do.
  • Major differences in the way designers created deliverables (fonts, screen size, fidelity, structure) made collaboration and handoff much more challenging.
This kind of thing happened way more often than it should have (which is never)

Despite the situation, there was also a lot of anxiety about how having a library could change the team’s workflow. The biggest worry was around losing creative freedom and limiting innovation, which design systems are often criticized for. These fears have prevented us from investing in design systems in the past, and we wanted to be mindful of them moving forward.

Library requirements

From this phase, a set of requirements for the library emerged:

  1. Support the main product families owned by the wearables team and make it easy to convert features between them
  2. Find ways for the library to provide guidance without being too restrictive
  3. Establish a regular update rhythm to stay up-to-date and actively engage with requests and feedback
  4. Include a process for auditing arbitrary differences and other weirdness

Establishing a fast, iterative process

We set up a system to handle component requests and met with the wearables team weekly to discuss them. To keep things moving, we kept the process light and prioritized iteration over perfection.

Auditing existing work

It didn’t take long for me to realize that the mockups and design specs linked in each ticket didn’t always match implementation. I started keeping a record of device screenshots to show the design team and help with auditing.

Through it, we caught several bugs and implementation errors to discuss with software. The compiled and organized screenshots also began to gain traction as an overview of the device for designers to reference.

The UX Audit file turned out to be helpful for newer designers learning about the device

Narrowing down device types

To feasibly build a single library to cover several product families, we decided go mid-fi and only include the details needed to make UX decisions. Out of the many product families and variations Garmin offers, we defined four basic wearable types that capture all of the products we needed. These would be designed at a similar screen size to make conversions between types faster and easier.

The basic device types and the products they cover

Designing for flexibility

We started with our most common interface element- list items. List items are notoriously complex with many different use cases and variations, so we used nested “building block” components to help split them into manageable pieces.

Since building blocks are controlled by instance swaps rather than variants, users would also be able to create entirely new kinds of secondary content or left/right elements and switch them in without breaking the original component. This gave us an incredibly flexible base that designers would have room to experiment with.

List items are built using modular pieces that can be swapped out

Built-in suggestions

Our approach was great at providing options and possibilities, but lacked direction. We also wanted to give users an easy path to the most common versions of an element, which is what they want 99% of the time anyway. This would be especially important to avoid overwhelming newer designers still getting the hang of our ecosystem.

This is where patterns, galleries, and templates come in.

Patterns act like presets for the most common variations of a base component, while galleries showcase all of the possible permutations supported by the component. Templates give designers a jump start on mocking up common interactions and demonstrate how the component should be used in context.

Patterns, galleries, and templates are published alongside the base component to help designers understand how the component works and get what they need at a glance

Intuitive Navigation

Nailing findability was personal to me.

There have been many times I’ve tried to search for something in a library, assumed it wasn’t supported, only to later learn it was- just under a name/category I didn’t think to try and never would have guessed.

Wearables lack standardized naming conventions for most UI elements, which makes it extra difficult to search for things by name. We looked into how naming works on software and learned that it’s also somewhat inconsistent there and would map poorly to our mental model of the device.

Our solution was to lean on visual search. We created a top-level page in the library that aggregates all of the supported templates and major screens on the device called the Component Gallery.

Previewing everything in the library on one page is doable for wearables since each screen is pretty small

The Component Gallery allows users to browse all of the screens we’ve built in one place without having to use Figma’s Asset Search or dig through individual library pages. This page also makes it easier for designers to see which components are already built and what still needs to be requested before opening a new ticket.

Reception

The Wearables Library made its official debut in early 2024 and only grew from there. Within a year, we had built 220 components with over 200,000 recorded inserts by 25 teams around the globe.

Numbers aside, part of what made working on this project so fun and rewarding was being able to see its impact on the wearables team.
